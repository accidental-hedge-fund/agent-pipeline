## Context

The pipeline advance loop dispatches stage handlers via a switch on the current stage label.
Today, `planning` and `plan-review` are handled by a special-case return:

```ts
case "planning":
case "plan-review":
  return {
    advanced: false,
    status: "waiting",
    reason: `${stage} is set mid-flight by the planning/plan-review handler; nothing to do at this point.`,
  };
```

This was correct when the only source of a `planning` label was an actively-running sibling
process. It is wrong when the sibling crashed before completing. The per-issue lock
(`/tmp/pipeline-{domain}-{N}.lock`) serializes runs: if the current process holds the lock
and sees `planning`, there is no sibling. The stranded label is the crash residue.

## Goals / Non-Goals

**Goals:**
- Eliminate the need for `pipeline triage N --stage ready` recovery when planning crashes.
- Surface a clear operator-visible diagnostic on crash recovery.
- Keep the implementation minimal: modify the dispatch switch only, reuse existing
  `transition()` and `planningStage.advance()` primitives.
- Maintain the existing concurrency guarantee: no two planning runs on the same issue.

**Non-Goals:**
- Resuming a partial planning artifact — a crashed run is restarted from scratch. Partial
  worktrees from the crashed run may exist on disk; `planningStage.advance()` handles that via
  its existing worktree-lookup and recreation logic.
- Distinguishing "crashed at step 2" from "crashed at step 5" — both are rolled back to
  `ready`. The granularity is not worth the added complexity.
- Changing `pipeline triage --stage` behavior or removing it — it remains a valid manual
  escape hatch.

## Decisions

**Decision: use the lock as the "no concurrent run" signal — no additional file check.**
The PID-based lock file (`lock.ts`) already handles stale-lock detection via `process.kill(pid, 0)`. By the time `dispatchStage()` is called, the lock has been acquired (or the process exited with "lock held by another process"). There is no additional check needed; the lock acquisition IS the proof. Adding a secondary "is planning active?" file would duplicate this mechanism and create a new failure mode (file not cleaned up).

**Decision: roll back to `ready` before calling `planningStage.advance()`.**
`planningStage.advance()` expects the issue to be on `ready`; it performs the `ready → planning` transition internally at step 2. Passing a stranded-`planning` or `plan-review` issue directly would make the transition `planning → planning` which is not a valid stage advancement and would fail or produce a confusing label state. Rolling back first keeps the planning stage handler's contract intact and avoids a second code path through `advance()`.

**Decision: treat `plan-review` the same as `planning` (roll back to `ready`).**
An alternative would be to resume the plan-review phase from where it was. However: (a) the plan artifact was generated by a crashed process and may be incomplete or uncommitted; (b) the plan-review reviewer prompt expects a freshly-generated plan in a specific format; (c) attempting selective resume adds significant complexity for an event that is rare. Restarting is safe, idempotent, and consistent with how other crash-recovery works in the pipeline.

**Decision: one-line diagnostic before rollback, not a comment on the issue.**
The recovery is transparent: the operator sees the console line, and the issue transitions cleanly back through `ready → planning → ...` with its own transition comment. Adding an additional GitHub comment ("recovered from crash") would create noise and is not needed for auditability — the `triage` sub-command already posts a transition comment.

**Decision: injectable deps seam for the new dispatch logic.**
The `dispatchStage()` function already receives `cfg` and `issueNumber`. The new code calls `transition()` (a `gh.ts` export) and `planningStage.advance()`. For unit tests, these calls must be fakeable. Following the existing `ShaGateDeps` / `AdvanceReviewDeps` pattern, the new logic SHALL be wrapped in a small `PlanningRecoveryDeps` interface injected with defaults, so tests can supply fakes without spawning real `gh` processes.

## Risks / Trade-offs

- *False recovery on a legitimate `plan-review` pause* — there is no legitimate pause on
  `plan-review`; the planning handler either completes the arc or crashes. The `waiting`
  behavior was always a placeholder, not a real wait-for-human signal. Risk: negligible.
- *Worktree from crashed run exists on disk* — `planningStage.advance()` calls `getForIssue()`
  or `getOnDiskForIssue()` which handles existing worktrees. No change needed here.
- *`transition()` from `plan-review` to `ready` is a backwards transition* — the pipeline's
  `transition()` function validates that the target is a recognized stage, but does not
  enforce forward-only ordering. Backward transitions are already used by `triage --stage`.
  Confirmed safe for this use case.
